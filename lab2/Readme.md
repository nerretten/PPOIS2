# Отчет по лабораторной работе №2

## Цель работы
* Изучить принципы построения графического пользовательского интерфейса.
* Разработать программную систему с графическим пользовательским интерфейсом на языке Python.

## Общее задание
Разработать оконное приложение с одним главным окном и несколькими дочерними диалогами, построенное по шаблону проектирования Model-View-Controller (MVC). 

**Основные требования к функционалу:**
1.  Формирование массива записей путем ввода через диалоговое окно.
2.  Поиск записей по заданным критериям с выводом результатов в отдельном диалоговом окне.
3.  Удаление записей по условиям через отдельное окно с уведомлением пользователя о количестве удаленных записей.
4.  Вывод записей (в главном окне и окне поиска) в постраничном виде (пагинация) с возможностью изменения количества записей на странице и навигацией по страницам.
5.  Сохранение и загрузка массива записей из XML-файла.
6.  Использование DOM-парсера для записи и SAX-парсера для чтения XML.

## Архитектура приложения (MVC)
Приложение строго разделено на три компонента:

### 1. Model (`lab2/models.py`)
Отвечает за бизнес-логику и хранение данных о студентах:
* Класс `Student`: Представляет структуру данных студента (ФИО, группа, типы пропусков: по болезни, другие, без уважительной причины). 
* Класс `StudentModel`: Управляет списком студентов. Реализует методы поиска (`search_students`) и удаления (`delete_student`).
* Для работы с XML используются встроенные библиотеки Python: `xml.dom.minidom` (метод `save_to_xml`) и `xml.sax` (метод `load_from_xml` с классом-обработчиком `StudentSaxHandler`).

### 2. View (`lab2/view/`)
Реализует графический интерфейс с использованием фреймворка PySide6:
* `main_window.py`: Главное окно (`MainWindow`) с таблицей (`QTableWidget`), панелью инструментов и элементами управления пагинацией.
* `dialogs.py`: Содержит классы диалоговых окон:
    * `AddStudentDialog`: Форма добавления студента с валидацией ввода.
    * `SearchDialog`: Окно поиска с динамически изменяемыми полями ввода (в зависимости от критерия) и встроенной таблицей с собственной пагинацией для отображения результатов.
    * `DeleteDialog`: Окно для ввода критериев удаления.

### 3. Controller (`lab2/controller.py`)
Связывает интерфейс и данные:
* Класс `PaginationManager`: Универсальный класс, управляющий отрисовкой конкретной страницы данных в любой переданной ему таблице. Обрабатывает нажатия кнопок навигации (`<<`, `<`, `>`, `>>`) и изменение размера страницы.
* Класс `Controller`: Инициализирует менеджеры пагинации, перехватывает сигналы (нажатия кнопок, выбор меню) из `View`, обращается к `Model` для обновления данных и передает новые данные обратно во `View`.

## Сценарии демонстрации системы

Для проверки корректности работы программы предусмотрены следующие сценарии тестирования (используются файлы `test_students.xml` и `test_students1.xml`, содержащие по 50 уникальных записей):

**Сценарий 1: Запуск и загрузка данных**
1. Открыть приложение. Убедиться, что таблица пуста.
2. В меню выбрать "Файл" -> "Открыть" и загрузить `test_students.xml`.
3. Убедиться, что таблица заполнилась. Строка статуса показывает общее количество записей (50) и количество страниц при текущих настройках (по умолчанию 10 на страницу).

**Сценарий 2: Тестирование пагинации**
1. Воспользоваться кнопками `>` и `<` для перелистывания страниц.
2. Нажать `>>` для перехода на последнюю страницу и `<<` для возврата на первую.
3. Изменить количество записей на странице (например, на 25). Убедиться, что таблица перестроилась (теперь 2 страницы).

**Сценарий 3: Добавление записи и валидация**
1. Нажать кнопку "Добавить".
2. Оставить поля пустыми и нажать "ОК". Убедиться, что появляется предупреждение об ошибке (срабатывает валидация).
3. Корректно заполнить все поля и сохранить. Убедиться, что счетчик "Всего записей" увеличился на 1, а студент появился в конце таблицы.

**Сценарий 4: Поиск по критериям**
1. Нажать кнопку "Поиск".
2. Выбрать критерий *"1. По номеру группы или фамилии"*. Ввести номер группы (например, `101`). Нажать "Найти". В нижней таблице должны отобразиться студенты только из этой группы (работает независимая пагинация результатов).
3. Выбрать критерий *"3. По фамилии или диапазону пропусков"*. Задать диапазон общего числа пропусков (например, от 10 до 20). Убедиться, что находятся студенты, у которых сумма всех типов пропусков (`all_skips`) попадает в этот диапазон. При этом выпадающий список типов пропуска должен быть скрыт.

**Сценарий 5: Удаление записей**
1. Нажать кнопку "Удалить".
2. Выбрать критерий *"2. По фамилии или виду пропуска"*. 
3. Выбрать вид пропуска "Без уважительной причины". Нажать "Удалить".
4. Убедиться, что появилось информационное окно с количеством удаленных записей (тех, у кого значение данного пропуска > 0).
5. Проверить главную таблицу: общее количество записей должно уменьшиться.

**Сценарий 6: Сохранение данных**
1. Выбрать "Файл" -> "Сохранить".
2. Указать новое имя файла (например, `output.xml`).
3. Убедиться, что файл успешно создан в директории, имеет корректный формат XML и содержит актуальное состояние таблицы.
